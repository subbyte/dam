# CLI

Last verified: 2026-07-02

## Overview

The `dam` CLI is a TypeScript Node package, distributed via npm, that users install on their own machine and point at a configured Platform deployment. It never runs inside the cluster. The current surface: `dam --version`, `dam --help` (built-in flags); `dam config set`; `dam ping`; `dam version`; the `dam auth` group (`login`, `logout`, `status`); the `dam agent` group (`list` [default], `get`, `create`, `create-interactive`, `delete`, `restart`); `dam chat`; `dam session list`; the `dam ssh` group (`connect`, `configure`, hidden `_proxy`); `dam template list`; `dam import`; the `dam file` group (`get`, `put`, `list`); the `dam network` group (`list`, `create`, `update`, `revoke`, `preset`, `apply-preset`, `trusted-hosts`); the `dam approval` group (`list`, `approve`, `deny`); the `dam connection` group (`list`, `templates`, `connect`, `disconnect`, `grant`, `revoke`, `update`); the `dam skill` group (`source list`, `source add`, `source remove`, `source refresh`, `catalog`, `list`, `install`, `uninstall`, `publish`); the `dam schedule` group (`list`, `get`, `create`, `update`, `enable`, `disable`, `delete`, `reset-session`); and the `dam channel` group (`available`, `list`, `slack connect`, `slack disconnect`, `telegram connect`, `telegram disconnect`, `allow`, `disallow`). Command groups are singular to align with `gh`, `git`, and `docker` conventions.

The CLI shares types directly with the api-server via a shared contract package, so server-side type changes reach the CLI without codegen or manual mirroring. Most routes are reached through plain HTTP calls against the api-server's tRPC endpoints; the `dam chat` verb additionally opens a WebSocket to the terminal relay for the interactive PTY session. The auth probes (`/api/auth/config`, OIDC discovery) stay as raw `fetch` because they are not tRPC.

## Trust boundary

The CLI runs on the user's machine. It reads and writes only under the XDG config and state directories (today: `config.toml` under `$XDG_CONFIG_HOME/dam/`; later, credentials under `$XDG_STATE_HOME/dam/`), and makes outbound network calls only to the configured server. There is no telemetry and no anonymous reporting — the platform collects nothing today and the CLI does not break that posture. The one deliberate exception is `dam ssh`'s editor modes (`-x code`, `-x zed`): dam's host entries live in its own `$XDG_CONFIG_HOME/dam/ssh_config`, and a single `Include` line is added to `~/.ssh/config` so the editor's SSH client resolves the agent host — see [SSH access](#ssh-access).

## Config

Two persistence concerns are split across the XDG directories: editable configuration (this file, under `$XDG_CONFIG_HOME/dam/`) and credentials (`auth.toml` under `$XDG_STATE_HOME/dam/`, written by `dam auth login`).

- **Location:** `$XDG_CONFIG_HOME/dam/config.toml` (default `~/.config/dam/config.toml`). Flat schema, no profile indirection.
- **Keys:** v0 has one — `server` (URL). Adding a new config key requires registering it at compile time — undeclared keys are a build error.
- **Precedence at resolve time:** flag (per-invocation `--server`, when commands grow one) > env var > file > error. There is no silent default.
- **Env var:** `DAM_SERVER` for the server URL (matches the `dam` binary name). Future keys follow the same `DAM_<KEY>` convention.
- **Writes:** read-merge-rename. The CLI never blows away unrelated top-level keys, so a user can hand-edit comments or future config knobs without losing them on the next `dam config set`.

## Compatibility negotiation

Before any networked verb runs, the CLI hits the api-server's unauthenticated `GET /api/version` (plain HTTP, outside the tRPC surface) to learn the server's version and the minimum CLI version it accepts. Three verdicts:

- **Ok** — local CLI is at or ahead of the server's reported version. Command proceeds.
- **BehindCurrent** — local CLI is below the server but at or above the floor. The CLI warns to stderr and proceeds (exit 0).
- **BelowFloor** — local CLI is below the server's `minClientVersion`. Gated verbs (see below) hard-fail with a non-zero exit; un-gated verbs surface the same verdict but proceed.

When no floor is configured (`minClientVersion` absent from the response), `BelowFloor` is never produced — the CLI proceeds with `Ok` or `BehindCurrent` as if the floor check were skipped.

The floor is configurable via Helm (`apiServer.minClientCliVersion`) so operators can drop support for known-broken older clients without rebuilding the image. `dam ping` and `dam auth login` opt into this gate explicitly; future networked verbs (`shell`, …) will too. `dam version` is the un-gated counterpart to `ping`: it surfaces the same verdict (and the same stderr warnings) but never refuses to run — it is informational, not gated, and always exits 0 even on probe failure.

## Authentication

`dam auth login` authenticates the user against the Active Host's Keycloak realm via the OAuth 2.0 Device Authorization Grant ([RFC 8628](https://datatracker.ietf.org/doc/html/rfc8628)). The realm advertises a public client `platform-cli` (no secret, device grant only) registered in the Helm chart; `/api/auth/config` exposes its id alongside the existing `platform-ui` id so the CLI never hardcodes it.

The flow:

1. Pre-flight `CompatService.check()` — same gate the `ping` verb uses.
2. `GET /api/auth/config` → `{ issuer, clientId, cliClientId }`.
3. `GET <issuer>/.well-known/openid-configuration` → device, token, revocation endpoints.
4. `POST <device endpoint>` → user code + verification URI; CLI prints the URI (and opens the browser unless `--no-browser`).
5. Polling `POST <token endpoint>` with `grant_type=urn:ietf:params:oauth:grant-type:device_code` per RFC 8628 §3.5 (slow_down → +5s, expired_token / access_denied → terminal).
6. On success, persist a per-host record into `$XDG_STATE_HOME/dam/auth.toml` (mode 0600, atomic tmp+rename, read-merge-write to preserve unrelated keys). If `--server` was supplied, persist it as the new active host in `config.toml`.

Credentials are keyed by host URL — the file is a `Map<HostUrl, HostAuth>` shape so switching between Platform deployments doesn't clobber state. `dam auth status` lists every host (active marked), the credential source, and the access-token expiry. **It never prints tokens.** `dam auth logout` best-effort RFC 7009 revokes the refresh token and atomically removes the host's entry — local clear always proceeds even when revocation fails (exit 0, stderr warning). Logout is not OIDC RP-Initiated Logout: the CLI must not kill SSO sessions for unrelated clients (the web UI, federated apps).

The `auth` module exposes a single application service — **`TokenProvider`** — which every future authenticated CLI verb consumes via `getValidAccessToken(host)`. Precedence: `DAM_TOKEN` env var (returned verbatim, no refresh) > Auth Store entry (refreshed proactively within 60s of expiry) > `not-logged-in` error. On a successful refresh the rotated refresh token is persisted atomically; on `invalid_grant` the host's entry is cleared and `session-expired` surfaces so the next command line directs the user to `dam auth login` again.

Those are the *client-side* signals. The server can also reject a request that looked valid locally, and the CLI surfaces that distinctly rather than as a transport failure: the tRPC client recognizes the api-server's raw `401`/`403` responses — alongside the existing `412` Terms gate — and maps them to typed errors. A `401` becomes the `not authenticated: session expired …` message; its hint is `run dam auth login` normally, but switches to "check `DAM_TOKEN` is valid and unexpired" when that env var is set — because a `DAM_TOKEN` bearer is supplied verbatim and never lives in `auth.toml`, so re-login can't fix it. A `403` becomes `access denied: <reason>` with no login hint, because the caller is authenticated but not authorized (e.g. a pending-approval account) and the remedy is to contact an admin, not to re-authenticate. Any other request the server reaches but rejects — a duplicate name, invalid input, an authorization failure on a scoped key — surfaces the server's own reason rather than `cannot reach server`, which is reserved for genuine connectivity failures that produce no HTTP response.

Concurrent writes to the auth store are not coordinated in v1. The store mutates `auth.toml` via read-merge-rename: the rename is atomic, but the surrounding sequence is not, so two `dam` processes that overlap (e.g. an interactive `dam auth login --server foo` running while a `TokenProvider` refresh for `bar` fires in another terminal) can each persist their own merged snapshot, and the later rename silently reverts the other host's entry. The failure surfaces later as an unexpected `session-expired` prompt — recoverable with `dam auth login`, but on a host the user may not remember touching. Same-host concurrent refreshes cost at most one forced re-login. A proper fix (per-host files or cross-process locking) is deferred — v1 targets solo, single-terminal use.

For headless / CI use, set `DAM_TOKEN=<bearer>` — the CLI uses it verbatim and bypasses `auth.toml`. There is no `--token` flag (avoids leaking tokens into shell history and `ps`). The variable accepts either a Keycloak access token or a Platform API key (`pk_…` prefix); the CLI does not branch — the server's bearer middleware dispatches by prefix.

### API keys (`dam auth token`)

`dam auth token` is the sub-tree that mints, lists, and revokes API keys. Three commands:

- **`dam auth token create --name <name> [--scope agents:read|agents:operate|agents:manage|credentials:read|credentials:manage…] [--agent <agent-id>…] [--expires <iso>] [--json]`** — calls `apiKeys.create` against the active host. The server returns the plaintext token *once*; by default the CLI prints it on stderr (so a `> file` redirect doesn't silently capture it) along with the cannot-be-recovered warning, and under `--json` emits the token on stdout inside the JSON payload for scripted capture. Default scope is `agents:operate`; default agent binding is `*` (every agent the owner owns now and in future). `agents:manage` keys must be unrestricted — the wildcard binding is required (per-agent management downscoping is a future refinement). API-key principals cannot mint other API keys — the management surface rejects them server-side.
- **`dam auth token list [--json]`** — emits id, name, scopes, agent binding, expiry, and last-used timestamp for every non-revoked key the caller owns. Plaintext is never displayed.
- **`dam auth token revoke <id>`** — soft-deletes by stamping `revoked_at`. The key is rejected on the very next request — the validator filters revoked rows at lookup time.

The sub-tree composes inside `auth/compose.ts` using the same TokenProvider the rest of the CLI consumes, with the same precedence: a `DAM_TOKEN` value wins over the stored interactive credentials. Minting with a `DAM_TOKEN`-supplied API key still fails — the server's management surface rejects API-key principals — but it fails server-side by design, not because the CLI filters the credential.

## Agent addressing

The `agent` module gives users a human-friendly path to address an Agent and exports the seam every Agent-targeted verb consumes.

- **Agent Ref** — what the user types. Either an Agent ID (anything starting with the Reserved ID Prefix `agent-`) or an Agent name. The split is syntactic; no probe disambiguates them.
- **Resolver policy** — `agent-…` is looked up via `agents.get`; anything else is matched by exact, case-sensitive name against `agents.list`. Zero matches → `not-found`; one → ok; two or more → `ambiguous`. No normalization. No retries. One round-trip per resolution in both branches.
- **Reserved ID Prefix** — the controller mints Agent IDs via `generateK8sName("agent")`. The api-server rejects Agent names beginning with `agent-` at create-time (zod refinement → BAD_REQUEST), eliminating the only ambiguous case.
- **Uniqueness** — `(owner, name)` is unique. Enforced at create-time as a list-then-check (TRPCError CONFLICT). The race window is accepted for CLI traffic; pre-existing duplicates fall through to the resolver's `ambiguous` path.
- **Resolver surface** — `AgentResolver` is exported from the `agent` module's `index.ts`. Downstream verbs (`dam chat`, `dam import`, `dam file`, …) import it from there and ask the module's compose for an `AgentService` bound to the resolved Active Host.
- **`EXIT_AGENT_NOT_RESOLVED = 5`** — single exit code shared by `not-found` and `ambiguous`; wrapper scripts don't need to branch on "did you mean a different one" vs "no such agent".
- **`--json` parity** — both `list` and `get` emit raw `Agent` / `Agent[]` from the contract. Empty list is `[]`, never `null`.

## Agent lifecycle

The CLI presents Agents as single, atomic entities — which they are: a single Agent resource carries both spec and runtime state, with no separate instance type.

- **Create** issues a single `agents.create` mutation. Env vars and description attach directly to the agent payload. There is no second hop and no rollback logic — the merged Agent CM is provisioned atomically server-side.
- **Delete** calls `agents.delete(id)`. The Kubernetes garbage collector cascades through the Agent CM's OwnerReferences to the StatefulSet, Service, NetworkPolicy, and any owned PVCs — the same flow the web UI's "delete agent" button uses. Delete confirms by default; `--yes` bypasses the prompt and is required on non-TTY stdin.
- **Restart** calls `agents.restart(id)`, which deletes pod-0 of the agent's StatefulSet. The controller recreates the pod with the current spec; persistent volumes (the home mount and any template-declared `persist: true` mounts) survive.
- **`--wait`** on `create` and `restart` polls `agents.get` every 2 seconds and settles on `state === "running"` (success) or `state === "error"` (terminal). `restart --wait` sleeps 2 seconds before the first poll so the controller has time to observe the pod deletion — otherwise the first poll might see stale `running` state from the doomed pod. Default timeout is 120 seconds; on timeout the agent is left as-is (no rollback) and the command exits non-zero. Under `--json`, every exit path emits a valid `Agent` payload on stdout: the success branch reuses the snapshot the wait loop already observed, and the timeout / non-wait branches refresh via `agents.get` with a fallback to the post-mutation snapshot if the refresh fails, so scripted callers never see empty stdout.

`dam template list` exposes the agent templates the operator has installed on the active host (the `claude-code`, `pi-agent`, etc. ConfigMaps the controller reads at boot). Templates are read-only from the CLI's perspective; operators add or remove them via Helm.

## Interactive agent setup

`dam agent create-interactive` is the interactive complement to `dam agent create`. It walks the user through name → template → model provider → optional GitHub PAT in a single TTY-bound flow and ends with the same agent + grants that the web UI's "Add agent" dialog produces. The scripted entry point is unchanged — `dam agent create <name> --template <id>` stays the path for shell scripts and CI, and `dam agent create-interactive` refuses to run when stdin is not a TTY (pointing the caller back at `agent create`).

Each step lines up with one server-side mutation, in the same order the web UI's `useCreateAgent` runs them:

1. **Model provider** — singleton per type (Anthropic, IBM LiteLLM, OpenAI, Bob), matching the web UI's provider cards. The picker lists existing provider Connections; "Add new..." creates a Connection (`connections.create`). If a Connection of the chosen type already exists, the flow offers to replace its credential in place (`connections.update`) instead of duplicating. For Anthropic the template (`anthropic` API key vs `anthropic-oauth`) is inferred from the pasted value; a replacement whose kind differs from the stored one is rejected rather than creating a second connection, preserving the singleton-per-type invariant (mirroring the UI's locked mode on edit). When the new Connection's template declares optional config inputs, the create path prompts each after the credential — skippable, validated against the input's declared spec — and sends them with `connections.create`; the replace path stays value-only (`connections.update` carries no config), matching the UI's edit form.
2. **GitHub PAT** *(optional)* — the picker lists existing `github-pat` Connections, plus "Add new..." and "Skip". A new PAT is a single `github-pat` Connection (`connections.create`; shape in [security-and-credentials.md](security-and-credentials.md#credential-storage)). If one already exists under the default name, the flow offers to replace its token in place (`connections.update`), defaulting to *not* replacing — overwriting a working token is destructive.
3. **`agents.create` → grant** — grants live on the agent, so the grant call has to come after `agents.create`. The picked Connections (the provider, plus the PAT when chosen) are granted via `connections.setAgentConnections` — an idempotent full replace, run inside a 5× / 2s retry to bridge the K8s-API visibility race for the just-created agent. The controller rolls the pod once when the grant lands; one rolling restart per agent is the cost.
4. **Wait for `running`** — same 120 s timeout the `agent create --wait` path uses. Timeout is success (the agent exists, the pod is slow); pod-failure state and Ctrl+C during the wait exit non-zero without rolling back.

Anything created during the run (new provider or GitHub-PAT Connection, the agent) is tracked in a small ledger (`newConnectionIds` plus the agent id); a failure in any of the post-prompt mutations triggers one cleanup pass — `agents.delete` cascade-tears-down the agent and its grants, then any new Connections are deleted by id (`connections.delete`). Picked-existing and replace-existing paths stay out of the ledger: the replace path overwrote the value in place, and the old value is unrecoverable. Anything the cleanup itself can't delete surfaces as an orphan summary so the user knows what to remove manually.

The post-success hint points at `dam chat <name>`. Interrupting at any prompt before the mutation chain exits cleanly with no orphans; interrupting during the wait leaves the agent in place with the same delete-hint, on the basis that the user chose to interrupt and the agent's existence is their call from there.

## Terminal attach

`dam chat <agent>` connects the user's local terminal to a running agent's interactive TUI over a WebSocket, using the same binary terminal frame protocol that the UI's terminal mode uses. The command requires an interactive TTY (stdin must be a TTY; piped input is rejected) and puts stdin into raw mode for the duration of the session so keystrokes — including Ctrl+C — pass through to the remote harness rather than being intercepted locally.

Three session strategies:

- **New** (default) — mints a fresh session id locally and connects; the PTY creates the session on attach. It surfaces in `session/list` with no `_meta` and decodes as terminal.
- **Continue** (`--continue`) — finds the most recent terminal-mode session for the agent. Errors if zero or more than one terminal session exists.
- **Resume** (`--resume <session-id>`) — targets a specific session by ID. If the target session is in chat mode, the CLI prompts the user to confirm a mode switch to terminal before proceeding; declining exits cleanly.

Strategy resolution happens **client-side**: sessions are agent-owned, so the CLI lists them over its own ACP connection to the api-server relay (decoding `_meta.platform`) and resolves the strategy locally — minting an id for `new`, matching terminal-mode sessions for `continue`, looking up the id for `resume` — then builds the terminal-relay URL itself. A confirmed chat→terminal switch is persisted over ACP (`session/resume` carrying `_meta.platform.mode`). There is no server-side session endpoint.

The `--reset` flag can combine with any strategy — it tells the api-server's terminal relay to kill the existing PTY and spawn a fresh one, which also triggers `resetSession` on the agent-runtime to close the agent-side ACP session and clear its log.

On disconnect, the CLI prints the session ID and a ready-to-paste `dam chat --resume` command so the user can reattach.

`dam session list <agent>` lists all sessions for an agent, showing session ID, mode, type, and creation time. The `--json` flag emits raw JSON for scripted consumption.

The chat module composes a per-host `SessionsPort` backed by a small ACP client speaking over the relay WebSocket that lists sessions and persists mode changes; agent resolution and auth still use the shared tRPC client. The terminal bridge owns the raw TTY ↔ WebSocket lifecycle — it receives the `terminalPath` the port built and constructs the full WebSocket URL locally, sending the auth token as a `token` query parameter. Both are injected through the module's compose root alongside the shared Token Provider and Agent Resolver seams.

## SSH access

`dam ssh` is a command group; `dam ssh connect <agent>` opens a standard SSH session to a running agent. Where `dam chat` attaches to a PTY running the harness TUI, `dam ssh connect` drops into a plain login shell (`/bin/bash`) in the agent's workspace and supports everything a real SSH server does — `scp`, `sftp`, port-forwarding, and VS Code Remote-SSH. The session inherits the agent's pod environment — the egress proxy (`HTTPS_PROXY`), credential sentinels, and `PATH` — so `git`, `gh`, and the AI CLIs work exactly as they do for the harness (the agent-runtime replays the live injected env through `~/.ssh/environment`, rebuilt per connection; without it sshd hands the shell a bare env and even DNS fails, since egress is proxy-only). The transport is an in-pod OpenSSH `sshd` spawned per-connection in inetd mode and tunneled over the agent WebSocket. The SSH wire protocol is opaque to the CLI and the api-server (which relays bytes verbatim) — SSH terminates at the in-pod sshd, which reuses the same WebSocket upgrade auth (token → ownership → terms) that `dam chat` does.

For `connect`, the client is chosen by a single `-x`/`--exec` flag, whose value is a binary name or path with an optional `:<mode>` suffix (`ssh` | `code` | `zed`) that forces how the binary is invoked. Without the suffix the mode is found by matching the binary's basename against each mode's known launchers — `code`/`code-insiders` → `code`, `zed` → `zed`. With no `--exec` at all the mode is `ssh` and the executable is `ssh`. A binary in none of the lists is disambiguated with the suffix (e.g. `-x my-build:code`).

Forms:

- **`dam ssh connect <agent>`** (mode `ssh`) — ensures the dam-managed keypair exists, then hands off to the `--exec` client (default `ssh`), configured with a `ProxyCommand` that re-invokes `dam ssh _proxy`. stdio is inherited, so the session is fully interactive.
- **`dam ssh connect -x code <agent>`** (or **`-x zed`**) — writes a managed `Host <alias>` block to dam's own `$XDG_CONFIG_HOME/dam/ssh_config` and ensures `~/.ssh/config` pulls it in via one `Include` line, then launches the editor against the agent's workspace via that alias. Mode `code` runs `<exec> --remote ssh-remote+<alias>` (exec defaulting to `code`); mode `zed` runs `<exec> ssh://agent@<alias>/…`. Both resolve the alias from `~/.ssh/config`, so they inherit the same ProxyCommand and identity. (`ssh` has no env-var override for its config location — only `Include`/`-F` — and the clients are often already-running singletons that ignore a launch-time `PATH`/`HOME`, so the `Include` is the reliable cross-client hook.) Mode `code` additionally pre-allows the VS Code Remote-SSH server-download hosts (`update.code.visualstudio.com`, `vscode.download.prss.microsoft.com`) as host-only egress rules before launching, so the agent's network gate doesn't pop an approval prompt in the web UI mid-connect — best-effort and idempotent (it skips hosts already ruled on and a wide-open preset), warning to stderr if it can't seed them.
- **`dam ssh configure <agent>`** (or **`--all`** / **`--clear`**) — writes the same managed `Host <alias>` block(s) and `Include` as the editor forms, then prints the alias and example `ssh`/`code`/`zed` invocations and exits without launching anything. `--all` reconciles the config to the active host's current agents — one `Host` block per agent, pruning blocks whose agent no longer exists; `--clear` removes every dam-managed block (deleting the config file and its `Include` once empty). For wiring up an editor or other tool by hand.
- **`dam ssh _proxy <agent>`** — hidden internal command: the `ProxyCommand` itself (not run by hand). It resolves the agent, obtains a token, registers the dam public key with the agent via `ssh.authorizeKey` (the agent-runtime tRPC route proxied at `/api/agents/<id>/trpc`), opens a WebSocket to the agent's `/api/ssh` relay, and shovels raw bytes between the socket and stdin/stdout. Only relayed bytes ever reach stdout (the SSH transport); diagnostics go to stderr.

Authentication is public-key: a dedicated dam keypair under `$XDG_STATE_HOME/dam/ssh/` (so the CLI never touches the user's own `~/.ssh` keys). Host-key checking is disabled (`UserKnownHostsFile=/dev/null`, `StrictHostKeyChecking=no`): the WebSocket upgrade is the real trust boundary and the stream is already TLS-encrypted, so the in-pod host key authenticates nothing extra and only broke connections when an agent PVC rotated it. The user's real identity is the bearer token verified at the upgrade; the SSH key is only the transport credential `ssh` requires. The `connect` and `configure` forms run the same compatibility gate as other networked verbs; `_proxy` resolves the server from config directly, since it runs once per SSH connection.

`dam ssh` depends on the OpenSSH client (`ssh`, `ssh-keygen`) on the user's machine; the editor modes additionally need the chosen client on PATH (`code`/`code-insiders` for mode `code`, `zed` for mode `zed`). Server-side, `sshd` ships in `platform-base`, so every agent image supports it; an image that stripped `sshd` from the base would reject the connection cleanly.

## Import

`dam import <agent-ref> <path...>` uploads one or more local files or folders into an Agent. The verb consumes the Token Provider seam from `auth`, the `createAgentResolver` factory from the `agent` module's `index.ts`, and the same `CompatService` / `ConfigService` gates every networked verb uses.

- **Wire shape** — POST `<server>/api/agents/<id>/import`, multipart/form-data with one part `bundle: application/gzip`. Bearer auth. Same contract the UI's [`importBundle`](../../packages/ui/src/modules/files/api/import-bundle.ts) targets.
- **Each path argument becomes one top-level entry** under `work/` on the Agent, named by its `basename(path)`. `dam import foo CLAUDE.md .claude src` lands at `work/CLAUDE.md`, `work/.claude/`, `work/src/`. The on-pod `finalize` ([packages/agent-runtime/src/modules/import/finalize.ts](../../packages/agent-runtime/src/modules/import/finalize.ts)) replaces each top-level entry atomically; other entries under `work/` are untouched.
- **Bundle** — a single gzipped tar built from the supplied paths with [`tar-stream`](https://github.com/mafintosh/tar-stream) and spooled to a tmpfile. Sent as a `FormData` `Blob` via Node's `openAsBlob` so undici computes `Content-Length` correctly (the server requires it). Symlinks anywhere in the imported tree are skipped (not followed). Excluded basenames at every level: `node_modules`, `.venv`, `__pycache__`, `.DS_Store` — same set the UI uses; rendered into `dam import --help` from the single source of truth.
- **Tmpdir lifecycle** — `$TMPDIR/dam-import-*` is created up front and torn down by an awaited `cleanup()` after upload. A SIGINT handler is installed alongside the tmpdir and removed by `cleanup()`; on Ctrl+C during pack or upload the handler does a synchronous best-effort rm and exits 130, since the default SIGINT termination skips `finally` blocks.
- **Top-level replace is destructive** at each named path. The CLI shows a TTY confirm prompt before any upload; `--yes` skips. Non-TTY without `--yes` refuses (exit 2). Cancelled-by-user prints `Cancelled.` to stdout (or `{"cancelled":true}` under `--json`), exit 0.
- **No service layer** — the verb has a single POST with status-based classification, so the action handler classifies inline (`switch (res.status)`) rather than introducing a port the way `agent` does for tRPC.

## Files

`dam file get <ref> <remote-path>` / `dam file put <ref> <local-path> <remote-path>` / `dam file list <ref> [<remote-path>]` are granular, non-destructive counterparts to `dam import`. Each verb operates on a single file in an Agent's workspace and consumes the existing api-server proxy at `/api/agents/<id>/trpc/files.*` — the same surface the web UI's file browser uses (see [packages/agent-runtime/src/modules/files.ts](../../packages/agent-runtime/src/modules/files.ts) for the underlying service, and [packages/api-server/src/apps/api-server/app.ts](../../packages/api-server/src/apps/api-server/app.ts) for the catch-all proxy that fronts it). No new server-side surface is introduced.

- **Wire shape** — a per-agent tRPC client typed against `agent-runtime-api`'s `AppRouter` and pointed at `<server>/api/agents/<id>/trpc`. Bearer auth via the shared Token Provider; ownership is verified at the proxy boundary in api-server. Verbs reuse the same `CompatService` / `ConfigService` gates and `AgentResolver` as every other Agent-targeted CLI verb.
- **`get`** — one `files.read.query({ path })`, decodes base64 for binary payloads or passes utf8 through for text, and writes to `basename(remote-path)` in cwd by default. `-o <local-path>` overrides the target; if it resolves to an existing directory the file lands inside as `basename(remote-path)` (cp-style). `--stdout` streams raw bytes (binary-safe). Refuses to clobber an existing local file unless `--overwrite` (the `--stdout` path bypasses the check). Files exceeding the 10 MB cap surface as `PAYLOAD_TOO_LARGE` from the route — the CLI exits non-zero with a hint that streaming for individual large files is not yet implemented.
- **`put`** — one `files.upload.mutate({ path, contentBase64, overwrite })`. Non-destructive by default; `--overwrite` opts into clobbering. Pre-flights the local file (must exist, must be a regular file) before any wire call. Refuses local directories — recursive upload is `dam import`'s job. The per-file size cap is enforced server-side and surfaces as `PAYLOAD_TOO_LARGE` from the route; the CLI doesn't duplicate the constant.
- **`list`** — one `files.tree.query()`. Default text output is one file path per line, directories omitted (pipe-friendly: `dam file list … | xargs -n1 dam file get …`). `--json` emits the full tree shape (files **and** directories with `type`). An optional `<remote-path>` argument prefix-filters the tree client-side.
- **Caps and exclusions** — the per-file size cap is enforced server-side in `agent-runtime`'s `FilesService` and surfaces on both `read` and `upload` as `PAYLOAD_TOO_LARGE` (single source of truth — the CLI doesn't duplicate the constant). Excluded basenames (`.git`, `.npm`, `.triggers`, `.claude.json`, `.initialized`, `node_modules`, `.DS_Store`) are invisible in `list` and refused on `put`. Symmetric to `dam import`'s exclusions.
- **No service layer** — each verb is one tRPC call; the action handler classifies inline rather than introducing a port, mirroring `dam import`.
- **Out of scope (v1)** — recursive directory download (combine `dam file list | xargs dam file get`), streaming for files > 10 MB, `dam file rm` / `mv` / `mkdir`, globs in `<remote-path>`. Each lands when a concrete use case appears.

## Network access

`dam network` manages per-Agent outbound-host rules — the pre-approvals that let an Agent reach external services without round-tripping the inbox. The CLI is a thin wrapper around the `egressRules.*` tRPC procedures, at parity with the per-agent network access editor in the UI.

Seven commands, agent-scoped ones positional-first on the agent ref to match the existing verb convention:

- `dam network list <agent>` — six-column table sorted by `(host, method, pathPattern)`; `--json` emits `EgressRuleView[]`.
- `dam network preset <agent>` — current effective preset (`none` / `trusted` / `all`).
- `dam network create <agent> --host <h> [--method] [--path] [--verdict]` — adds a manual rule. `--method` and `--path` default to `*` (the L4 host-only rule); `--verdict` defaults to `allow`.
- `dam network update <rule-id> [--method] [--path] [--verdict]` — partial update; at least one flag is required. Flips `source` to `manual` server-side.
- `dam network revoke <rule-id>` — deletes the rule. Idempotent — unknown IDs exit 0.
- `dam network apply-preset <agent> --preset <name>` — bulk-seeds; replaces existing `preset:*` rows; preserves `manual` and `connection:*` rows.
- `dam network trusted-hosts` — the platform-wide hosts seeded by the `trusted` preset.

Two pieces of non-obvious behavior the CLI surfaces:

- **Path/method-specific rules require L7 enforcement.** When `create` or `update` produces a rule with non-wildcard method or path, the gateway has to re-MITM the host, which rolls the Agent pod (~5–15s). The CLI prompts on TTY; `--yes` bypasses; non-TTY without `--yes` exits `EXIT_INVALID_INPUT` (2). The check fires on user-passed flags only — false positives for already-L7 rules are accepted in preference to fetching the existing row.
- **Preset `all` is a development escape hatch.** `apply-preset --preset all` confirms before applying (mirrors the UI's `window.confirm`). Every preset apply prints an explanatory line to stderr that manual and connection-derived rules are preserved.

`update` against an unknown rule ID exits `EXIT_RULE_NOT_FOUND` (6) — the contract change in `egress-rules-service.ts` converts the plain `Error("egress rule not found")` to `TRPCError({ code: "NOT_FOUND" })` so the CLI can classify the error via `data.code`.

The `source` field is rendered via the shared `formatEgressRuleSource` helper in [packages/api-server-api/src/modules/egress-rules/format.ts](../../packages/api-server-api/src/modules/egress-rules/format.ts) so the CLI and UI use identical labels.

## Approvals

`dam approval` actions the unified HITL inbox — the pending decisions Agents are waiting on, the same `pending_approvals` queue the web UI surfaces. The boundary with `dam network`: `network` manages egress **rules**, the pre-approvals that stop a prompt from ever appearing; `approval` actions the **prompts that did appear**. A durable approve on a network (`ext_authz`) request writes a rule that then shows up in `dam network list <agent>` with source "from inbox". Tool-call (`acp_native`) verdicts go to the harness, which owns its own permission persistence — no egress rule is ever written for them, including under `--entire-host` (the server falls back to the durable approve).

Three verbs, a thin client over the `approvals.*` tRPC procedures:

- `dam approval list [<agent>]` — the inbox, owner-wide by default; the optional Agent Ref scopes to one Agent. Six-column table (`ID TYPE AGENT REQUEST STATUS EXPIRES`); `REQUEST` renders via the shared `describeApprovalPayload` helper in [packages/api-server-api/src/modules/approvals/format.ts](../../packages/api-server-api/src/modules/approvals/format.ts) so CLI and UI labels can never drift, and is the flex column — collapsed to one line and truncated to the terminal width (use `--json` for the untruncated text). `--status pending|resolved|expired|all` defaults to `pending` — the actionable set, a deliberate divergence from the UI's unfiltered inbox; `all` omits the filter. `--limit <n>` accepts 1–500 (the contract schema rejects more; the CLI validates client-side and exits 2). Rows come back newest-first. `--json` emits raw `ApprovalView[]`.
- `dam approval approve <id> [--once|--entire-host]` — bare `approve` is the durable verdict (`approvePermanent`): for `ext_authz` it writes a permanent allow rule and the success line names it. `--once` (`approveOnce`) releases only the held call; `--entire-host` (`approveHost`) writes an allow rule covering the entire host `(host, *, *)`. `--once` and `--entire-host` together exit `EXIT_INVALID_INPUT` (2).
- `dam approval deny <id> [--once]` — bare `deny` is durable (`denyForever`, writes a deny rule for `ext_authz`); `--once` (`dismiss`) denies only the held call, so the same shape re-prompts.

The verdict mutations return an **Action Outcome** — the post-CAS truth — and the exit code derives from it: `applied` and `rule_written_expired` (the hold already lapsed but the durable rule was written, so the next request of that shape resolves from it) exit 0, with the human output naming any rule written; `not_actionable` (unknown, foreign, or already-settled id — deliberately indistinguishable) prints to stderr and exits **`EXIT_APPROVAL_NOT_ACTIONABLE` (8)**, so a typo'd id can't report success. The outcome is computed server-side and is race-free where a CLI-side pre-resolve would not be. `--json` emits the raw outcome and keeps the same exit codes. An `acp_native` verdict is delivered, not resumed — the CLI records the decision; real-time unblock of the held turn is best-effort (live relay or outbox retry, server-side).

## Connections

`dam connection` brings the CLI to parity with the UI for managing **Connections** — the stored credentials and contributions an Agent uses to reach authenticated external APIs and MCP servers (the concept is owned by [connections.md](connections.md)). The CLI is a thin client over the `connections.*` tRPC procedures. Seven commands:

- `dam connection list [<agent>]` — without an Agent, a five-column table (`ID NAME CATEGORY STATUS HOSTS`) of every team connection — apps and MCP servers alike — sorted by `(category, name, id)`; with an Agent, only that Agent's granted connections. `--json` emits the raw `ConnectionView[]`.
- `dam connection templates` — a read-only listing of the connection catalog a Connection can be created *from* (apps + custom-credential types), as a five-column table (`ID NAME CATEGORY AUTH DESCRIPTION`) sorted by `(category, name, id)`; `--json` emits the raw `ConnectionTemplateView[]`. It excludes MCP-category templates — every listed id is a valid `connect <id>` argument, and `connect` rejects raw mcp-category ids in favor of a URL — and footers the MCP-by-URL path to stderr. Distinct from `connection list`: that shows the connections you've *created*; this shows the ones you *can* create.
- `dam connection connect <provider-or-url>` — create a Connection. The positional is either a **provider template id** or an **MCP server URL** (anything that parses as an `http(s)` URL); template ids are slugs that never contain `://`, so the form is unambiguous.
  - **Template id.** Resolves the template by id (unknown id exits `EXIT_INVALID_INPUT`; a raw MCP-category template id is rejected with a "pass its URL" redirect), collects the inputs the template declares (typed flags such as `--client-id`/`--value`, with interactive `@clack` prompts filling any *required* gap on a TTY — secrets masked — and a non-TTY gap erroring out naming the missing flag). **Overridable inputs** — those with a server-side preset, either an operator-baked default or a client credential inherited from a sibling connection in the same family (e.g. Gmail reusing a Google Drive client app) — are never prompted: omit the flag and the server applies the preset, or pass the flag (`--client-id`, `--client-secret`, `--app-slug`, …) to override it with your own, reaching parity with the web UI's "Customize defaults". When a preset fills an input you weren't asked about, `connect` prints a generic stderr note naming the field(s) and the override flag(s) — generic because the CLI can't distinguish an operator default from a family inheritance (#554). A template's optional **config inputs** (e.g. Bob's model and tenant scoping) are supplied non-interactively via a repeatable `-c, --config key=value` flag — keys must name a declared input, values are validated against its spec, blanks skip and duplicates last-win; they are never prompted, keeping the scripted path flag-only.
  - **MCP server URL.** Discovers the server's auth requirement via `connections.discoverMcp` (overridable with `--auth oauth|none`; an invalid value exits `EXIT_INVALID_INPUT`), picks the `custom-mcp-oauth` or `custom-mcp-none` built-in template accordingly, and derives the connection name from the host (`mcp.notion.com` → `notion`; `--name` overrides, and a TTY offers the derived value as an editable default).
  - Both forms converge on `connections.create`, then the shared tail: for an OAuth template it `connections.startOAuth` → opens the system browser (`--no-browser` prints the URL instead; an open failure degrades to printing) → polls `connections.get` every 2s up to `--timeout` (default 300s, under the server's 10-min pending-flow TTL) until `status === "active"`. Header/none templates (including a no-auth MCP server) need no browser and report success immediately. A poll timeout exits non-zero with a "check `dam connection list`" hint and leaves the connection in place (it may complete late). `--json` emits `{ ok, id, status, authKind }` (plus `presetsApplied` — the input names a preset filled — whenever a preset applied, and `authUrl` under `--no-browser`); under `--json` the human preset note is suppressed in favor of that field.
- `dam connection grant <agent> --connection <id-or-name>…` — grant one or more connections to an Agent, each addressed by id or unique name. Additive.
- `dam connection revoke <agent> --connection <id-or-name>…` — revoke one or more connections from an Agent, each addressed by id or unique name. Idempotent.
- `dam connection disconnect <id-or-name>` — delete a team connection (by id or unique name) and its stored credential.
- `dam connection update <id-or-name> [--value <secret>]` — rotate a header-auth connection's stored credential in place (`connections.update`), preserving its existing agent grants, identity, and contributions. The value comes from `--value` or a masked prompt on a TTY (a non-TTY without `--value` exits `EXIT_INVALID_INPUT`). Header-auth only: OAuth connections re-establish their credential through the browser flow, so a non-header connection is rejected (`EXIT_INVALID_INPUT`) before any mutation. `--json` emits `{ ok, id, name }`.

Everything is tRPC (`connections.list`, `listTemplates`, `create`, `update`, `startOAuth`, `discoverMcp`, `get`, `getAgentConnections`, `setAgentConnections`, `delete`) — there is no REST surface and no service state beyond the per-host client. Four pieces of non-obvious behavior:

- **Refs are id-or-name; resolution happens client-side.** `grant`/`revoke`/`disconnect` accept either a `conn-…` id or a connection's name. Names are unique per owner (DB-enforced), so a name resolves to at most one connection — no ambiguity, unlike Agent refs. The shared `resolveConnectionRef` helper does the lookup against `connections.list`: a `conn-…` prefix matches by id, anything else by name.
- **Grant/revoke are read-merge-set; disconnect resolves before deleting.** `connections.setAgentConnections` is a full replace, so the CLI reads the Agent's current grants (`getAgentConnections`), unions in (grant) or subtracts out (revoke) the resolved ids, and writes the whole set back — letting the server re-fan-out each connection's contributions (env vars + egress injection) for the correct final set. `grant` resolves every ref against the team list and rejects any that don't (`EXIT_INVALID_INPUT`) so a dead grant is never stored. `revoke` resolves names too, but lets a raw `conn-…` id pass through even when it's no longer in the team list — that's how a stale grant (a since-deleted connection) is cleaned up; only an unknown *name* errors. `disconnect` resolves its ref against the team list and exits `EXIT_INVALID_INPUT` when nothing matches: the server's `delete` is idempotent (an unknown id still succeeds), so resolving first is what prevents a typo'd id or a bare name from reporting a false "Disconnected".
- **Stale grants are surfaced, not hidden.** `list <agent>` intersects the Agent's granted ids with the team list; any granted id that no longer resolves to a live connection is reported on stderr rather than rendered as a fabricated row.
- **`connect` polls a server-side completion marker, not token expiry.** The browser OAuth flow is server-mediated (the api-server is the OAuth client; the provider redirects to the server's fixed callback, which redirects on to the *web UI's connections page* (`/connections`) — there is no terminal callback), so the CLI can only observe completion *through the server* by polling `connections.get(id).status`. That status now derives from a `connectedAt` marker the server stamps on every successful token exchange — independent of whether the provider returns a token expiry — so a connection with a non-expiring token (e.g. classic GitHub OAuth) reaches `active` rather than sticking at `pending`. (Before this marker existed, expiry was the only signal, so such connections never read as `active` — in the CLI poll or the UI list.)

MCP servers connect via their URL (`dam connection connect https://…`), which discovers auth and selects the right built-in MCP template. `dam connection templates` lists the *connectable* catalog but deliberately excludes mcp-category templates (`connect` rejects their raw ids), so the URL-first form remains how MCP servers are added — the templates listing footers exactly that. Listing, granting, revoking, and disconnecting are category-agnostic and unchanged — an MCP connection shows up in `list` with `CATEGORY=mcp` and is removed by `disconnect <id>` like any other.

## Channels

`dam channel` brings the CLI to parity with the UI's channels panel — the messenger surfaces (Slack, Telegram) that let users drive an Agent from outside the platform UI. The channel model — a platform-level Slack app the operator configures once via Helm versus a per-Agent Telegram bot whose token the owner supplies — is owned by [channels.md](channels.md); a binding is 1:1 with an Agent.

**The bindings live on `agents.*`, not `channels.*`.** Only `channels.available` is a channel route; every binding is an `agents.*` mutation and the channel list is read off the Agent itself (`AgentView.channels` / `.allowedUserEmails`), so there is no dedicated channels-list route. The `dam channel` group is the user-facing seam over those agent routes. Two read verbs today:

- `dam channel available` — the messenger providers the operator enabled on this host, a per-provider capability flag (`channels.available` → `{ slack?, telegram? }`). Human output lists the enabled providers; an empty/all-false result prints "No messenger channels enabled on this host." `--json` emits the raw object.
- `dam channel list <agent>` — the Agent's connected channels as a `TYPE IDENTIFIER` table (Slack rows show the channel id; Telegram shows no identifier because its bot token is write-only server-side), followed by the Agent's Slack allow-list (one `allowedUserEmails` entry per line, or `(unrestricted)` when empty). `--json` emits the composite `{ channels, allowedUserEmails }` — the allow-list rides here because the allow/disallow verbs don't print it. The Agent Ref resolves through the shared `AgentResolver` (`EXIT_AGENT_NOT_RESOLVED` (5)).

The bindings themselves are provider sub-groups — `dam channel slack` and `dam channel telegram` — so each provider keeps its own flags and help. The two providers diverge by design (see [channels.md](channels.md)): Slack is a platform-configured app (the user supplies only a channel id, no OAuth round-trip), Telegram is a per-Agent bot (the user supplies a `@BotFather` token stored write-only in a k8s Secret). Each connect/disconnect verb emits the Agent's updated `ChannelConfig[]` under `--json`.

- `dam channel slack connect <agent> --channel-id <C0…>` — binds a Slack channel (`agents.connectSlack`). A success hint reminds the user that drivers must run `/platform login` inside Slack — the CLI can't drive that identity-link step. One Slack channel binds to one Agent: a channel already bound to a *different* Agent returns `CONFLICT` server-side, which the CLI surfaces as "already bound to another agent" and exits `EXIT_INVALID_INPUT` (2).
- `dam channel slack disconnect <agent>` — unbinds (`agents.disconnectSlack`). Idempotent: a not-connected Agent is a server-side no-op success.
- `dam channel telegram connect <agent> [--bot-token <t>]` — binds a Telegram bot (`agents.connectTelegram`). The token is masked-prompted (`@clack` `password()`) when omitted on a TTY; omitted on a non-TTY it exits `EXIT_INVALID_INPUT` (2) naming the missing flag. The token is never echoed back, and `list` afterwards shows `telegram` with no identifier.
- `dam channel telegram disconnect <agent>` — unbinds and deletes the per-Agent k8s Secret (`agents.disconnectTelegram`). Idempotent.

Both connect verbs **precheck `channels.available`** and exit `EXIT_INVALID_INPUT` (2) early — with the operator-facing message the server's `PRECONDITION_FAILED` would carry (`Slack app token not configured` / `Telegram channel not enabled`) — rather than letting a connect to a disabled provider round-trip.

The agent-wide allowed-user list — the emails (besides the owner) permitted to drive the Agent from a connected channel — is managed by a paired `allow` / `disallow`, mirroring `connection grant`/`revoke`. It **gates Slack only** (empty = unrestricted, non-empty = an allow-list); Telegram ignores it (per-thread `/login`, no workspace identity — see [channels.md](channels.md)).

- `dam channel allow <agent> --user <email>…` / `dam channel disallow <agent> --user <email>…` — add or remove emails. The field (`Agent.allowedUserEmails`) is set through `agents.update`, which **full-replaces** the array, so the CLI does read-merge-write: it reads the resolved Agent's current list, unions in (`allow`) or subtracts out (`disallow`) the `--user` values, and writes the whole array back. The merge is a pure helper in the **agent** module (`mergeAllowedUserEmails`) so a future `dam agent update` reuses one copy. Both are idempotent. `allow` validates each email client-side (`z.email()`) before any wire call and exits `EXIT_INVALID_INPUT` (2) on a bad value; `disallow` matches by exact string, so it doesn't. `--json` emits `{ ok, agentId, allowedUserEmails }`. Listing the allow-list is `dam channel list`'s job — these verbs don't print it.

## Skills

`dam skill` brings the CLI to parity with the UI's Skills panel for the **source management**, **core install/use**, and **publish** workflows — registering and refreshing sources, browsing a source's catalog, seeing what's installed on an Agent (with drift), installing/uninstalling, and publishing a standalone skill upstream as a GitHub PR. The concept (sources, scanned skills, installed refs, drift, content hashes) is owned by [skills.md](skills.md). The CLI is a thin client over the `skills.*` tRPC procedures; the only server-side change behind it is hardening `sources.create` to return `CONFLICT` on a duplicate so `add` renders a clean "already registered" message. Nine verbs:

- `dam skill source list [--agent <ref>]` — registered sources, a four-column table (`ID NAME GIT URL KIND`) with `KIND` ∈ {User, Platform, Agent}; `--agent` also includes the Agent template's declared sources. `--json` emits the raw `SkillSource[]`.
- `dam skill source add <git-url> [--name <n>]` — registers a User source. The name defaults to the URL path (`github.com/acme/skills` → `acme/skills`; slash kept, not slugified — skill-source names allow it); `--name` overrides, and a TTY offers the derived value as an editable default. Adding a URL a Platform/Agent source already uses **shadows** it — removing your source re-exposes the original. An invalid URL or an already-registered one exits `EXIT_INVALID_INPUT` (2). `--json` emits the raw created `SkillSource`.
- `dam skill source remove <source>` — untracks a User source (`<source>` is an id or git URL, the same resolver as `catalog`). It is **untrack, not uninstall**: the source's `agent_skills` tracking rows are dropped across all your agents, but the skill files stay installed on the pods (they become standalone) — there is no `--purge`. Platform and Agent sources are rejected client-side. Confirms on TTY (the prompt says the files stay), `-y/--yes` skips, non-TTY without `--yes` refuses. `--json` emits `{ removed, id, name }` or `{ cancelled: true }`.
- `dam skill source refresh <source>` — drops the source's upstream scan cache so the next `catalog` re-scans. No confirmation; allowed on any resolved source, including Platform. It resolves against User + Platform sources only, so a template (`Agent`) source isn't reachable here — scan one via `catalog <source> --agent <ref>`. `--json` emits `{ refreshed, id, name }`.
- `dam skill catalog <source> [--agent <ref>]` — scans one source. Without `--agent`, `NAME DESCRIPTION`; with `--agent`, a `STATUS` column annotates each skill `not-installed` / `installed` / `update-available`. `--json` without `--agent` is the raw `Skill[]`; **with `--agent` each row is augmented with a CLI-computed `status` field** — the one documented departure from raw-passthrough parity, since the annotation is the whole point of `--agent` and re-deriving it would force a second call.
- `dam skill list <agent>` — two sections: **Installed** (`SOURCE NAME VERSION`, git URLs resolved to source names) and **Standalone** (`NAME PUBLISHED`, where `PUBLISHED` is the PR URL from the skill's most-recent publish record, else `—`); each section is omitted when empty. `--json` emits the full `skills.state` object `{ installed, standalone, instancePublishes }` (a deliberate change from the previous `SkillRef[]`).
- `dam skill install <agent> --source <id|url> --name <skill>` — installs at the source's current HEAD.
- `dam skill uninstall <agent> --source <id|url> --name <skill>` — removes it. Idempotent.
- `dam skill publish <agent> --name <skill> --source <id|url> [--title <t>] [--body <b>]` — opens a GitHub PR lifting a **standalone** (locally-authored) skill upstream to the **target** GitHub source. On a TTY, prompts for title/body with defaults; flags or a non-TTY pass through to the server defaults. `--json` emits `{ prUrl, branch }`.

Seven pieces of non-obvious behavior the CLI surfaces:

- **Two source identifiers.** `sources.list` and the scan (`skills.list`) key on the source **id**; `install`/`uninstall` key on the source's **git URL**. So `--source <id|url>` is resolved client-side against `sources.list` — a value containing `://` matches by `gitUrl`, anything else by `id` (no name resolution; source names allow spaces) — mirroring `resolveConnectionRef`. Resolving first both confirms the source is **registered** (the scan does `getSource` → `NOT_FOUND` otherwise; there is no unregistered-URL path) and yields the `id` the scan needs and the `gitUrl` the mutations need.
- **Install scans first; install == update.** The `install` contract requires `version` (a HEAD SHA) + optional `contentHash`, but there is **no `--version` flag** — `install` scans the source first, finds the named skill, and passes its HEAD `version`/`contentHash` through, exactly as the UI does. With no SHA pinning, re-running `install` rebakes to HEAD — which *is* the UI's "Update" button — so there is **no separate `update` verb**. `uninstall` needs no scan; it keys on `(git URL, name)`.
- **Drift only in `catalog --agent`.** Drift = installed `contentHash` ≠ HEAD `contentHash`, which requires a scan, so the `update-available` annotation surfaces only where the scan already happens. `list <agent>` is scan-free and never annotates — it would otherwise have to scan N sources to label an inventory.
- **Server-side wake; the CLI surfaces only the residual.** After [#670](https://github.com/dam-agents/dam/issues/670), `install`/`uninstall` and the private/non-GitHub scan route through the server's shared pod-reachability primitive, so a hibernated Agent **wakes transparently** server-side — the CLI never special-cases hibernation. It surfaces only the residual failure: an Agent stuck in an error state or a wake-to-ready timeout exits **`EXIT_AGENT_NOT_REACHABLE` (7)**. The server signals the relevant conditions through two tRPC error codes — `PRECONDITION_FAILED` (a private source scanned with no pod, an error-state Agent, or GitHub refusing) and `INTERNAL_SERVER_ERROR` (the wake-to-ready timeout) — and the CLI disambiguates them client-side. **With no agentId** (a bare `catalog <private-source>`) it means "this source needs a pod to scan" → `EXIT_INVALID_INPUT` (2) with a "pass `--agent`" hint. **With an agentId**, a `platform-cta:<url>` marker in the message means the scan reached the pod but GitHub refused (app not connected / access restricted) — the CLI strips the marker, prints the message plus a "connect the source — `<url>`" hint, and exits `EXIT_INVALID_INPUT` (2), mirroring the UI's connect CTA; without that marker it means the pod couldn't be woken → exit 7. `list <agent>` reads `skills.state`, which **degrades to last-known installed** when the pod is down and never wakes it, so the inventory verb never hits exit 7.
- **Publish keys on the source `id`, guards GitHub up front, and surfaces relayed failures precisely.** Unlike `install`/`uninstall` (which key on the git URL), `publish` passes the resolved source's **`id`** as `sourceId`. The target must be a GitHub source: a non-GitHub `--source` (`canPublish !== true`) is rejected client-side **before any publish call** → `EXIT_INVALID_INPUT` (2). Auth/authz is **not** preflighted — GitHub-not-connected, agent-not-granted, and repo-not-allowlisted all surface at publish time as a `platform-cta:<url>` marker; the CLI strips it, prints the message plus `Fix: <url>` (and `{ error, fix }` under `--json`), and exits 2, mirroring the UI's "Fix it" link. The pod is woken server-side ([#670](https://github.com/dam-agents/dam/issues/670) covers publish), so the CLI surfaces only the residual: an error-state Agent or wake-to-ready timeout exits **7**, while a missing skill or a GitHub 4xx/5xx prints the relayed server message verbatim and exits **1** (not the "cannot reach server" wording). Title and body default server-side (`Add <name> skill` / a brand blurb) and are prompted on a TTY.
- **Vocabulary: `catalog` (skills) vs `templates` (connections).** Both are "the thing a source/catalog exposes", but the nouns differ deliberately — a connection "template" is an instantiable domain noun (`ConnectionTemplateView`), whereas a skill catalog is just what a source exposes. The shared rule across the CLI: `list` = *your* resources; a different verb names the upstream catalog.
- **Source mutation resolves first, and every "can't do that" case is exit 2.** `remove`/`refresh` resolve `<source>` against `sources.list` before mutating — the server's `delete` is a silent no-op on an unknown id, so resolving first stops a typo from reporting a false success. An unknown ref, a Platform/Agent source handed to `remove`, and a `refresh` that raced a delete all exit `EXIT_INVALID_INPUT` (2); there is no source-specific exit code.

Everything is tRPC (`skills.sources.list`, `skills.sources.create`, `skills.sources.delete`, `skills.sources.refresh`, `skills.list`, `skills.state`, `skills.install`, `skills.uninstall`, `skills.publish`) — no REST surface and no service state beyond the per-host client. Agent refs (the `<agent>` positional and the `--agent` flag) resolve through the shared `AgentResolver`, so an unresolvable ref exits `EXIT_AGENT_NOT_RESOLVED` (5) like every other Agent-targeted verb.

## Schedules

`dam schedule` brings the CLI to parity with the UI's Schedules panel — the time-triggered tasks attached to an Agent. A schedule fires a task prompt on an RFC 5545 RRULE recurrence in a chosen IANA timezone, with optional first-class quiet-hour windows whose covered occurrences are dropped rather than deferred, and runs each tick either as a fresh session or one continuous resumed session. Unlike skills, schedule CRUD is **pure tRPC/DB** — there is no running pod, no reachability/wake: the controller fires schedules out-of-band, and the CLI only reads and mutates the spec. The one exception is `reset-session`, which enqueues a durable runtime-outbox event (the schedule's accumulated session is agent-owned) that lands even with the pod scaled to zero. The CLI is a thin client over the `schedules.*` procedures. Eight verbs, agent-scoped ones positional-first on the agent ref, schedule-targeted ones on an opaque schedule id:

- `dam schedule list <agent>` — a seven-column table (`ID NAME RECURRENCE TZ ENABLED NEXT-RUN LAST-RESULT`); RRULE rows render the recurrence human-readable, a legacy cron row renders its raw cron string, and an agent-authored schedule gets an `(agent)` marker on NAME. `--json` emits the raw `Schedule[]` view (`[]` when empty).
- `dam schedule get <schedule-id>` — one schedule, expanded (recurrence text + raw RRULE, timezone, each quiet window with a `(disabled)` marker, session mode, `createdBy`, status); a cron schedule shows its raw string instead. `--json` emits the raw view.
- `dam schedule create <agent> --name --task <recurrence> [--timezone] [--quiet-window …] [--session-mode]` — creates an RRULE schedule.
- `dam schedule update <schedule-id> [--name] [<recurrence>] [--timezone] [--quiet-window …] [--task] [--session-mode]` — partial update; at least one flag is required.
- `dam schedule enable <schedule-id>` / `dam schedule disable <schedule-id>` — idempotent.
- `dam schedule delete <schedule-id>` — confirms on TTY; `--yes` bypasses; non-TTY without `--yes` refuses. Idempotent.
- `dam schedule reset-session <schedule-id>` — clears a continuous schedule's accumulated session so the next tick starts fresh.

`<recurrence>` is exactly one of `--daily HH:MM`, `--every <Nm|Nh>`, or `--rrule <RFC 5545 body>`, with an optional `--weekdays MO,WE,FR` BYDAY filter on the first two. Six pieces of non-obvious behavior the CLI surfaces:

- **Recurrence flags compile to RRULE client-side via the shared builder — the CLI never assembles an RRULE itself.** The preset→RRULE compile (`buildRRule`), the human-readable `rruleToText`, the host-zone `detectTimezone` default, and the `hasVisibleOccurrence` precheck are the same React-free functions the UI's form uses, extracted into the contract package ([`api-server-api/src/modules/schedules/rrule.ts`](../../packages/api-server-api/src/modules/schedules/rrule.ts)) so the UI, the CLI, and the api-server share one copy. `--weekdays` is rejected with `--rrule` (a raw body carries its own BYDAY).
- **`update` is read-merge-write of the whole spec.** `schedules.updateRRule` requires the complete payload (`name + rrule + timezone + quietHours + task`), so the CLI fetches the current schedule, overlays the provided flags, and sends the full spec — mirroring `connection grant/revoke`. Omitted flags keep their current value; with no recurrence flag the existing rrule is preserved verbatim. A recurrence is one atomic unit: any of `--daily/--every/--rrule` rebuilds the whole rrule, and rebuilding a preset recurrence *without* `--weekdays` prints a stderr note when it silently clears a prior BYDAY filter. `--quiet-window` replaces the entire window set.
- **`update` is RRULE-only; legacy cron schedules are read-only.** `list`/`get`/`enable`/`disable`/`delete` operate on any schedule, but `update` against a cron schedule exits `EXIT_INVALID_INPUT` (2) with a "recreate it as an RRULE to edit" hint. (The view is not a discriminated union — the CLI tells the two apart on the nullable `rrule` field, not on `type`.)
- **Validation is authoritative server-side; the CLI prechecks only `hasVisibleOccurrence`.** Before any wire call, `create`/`update` refuse a schedule whose every occurrence falls inside a quiet window (the "would never fire" guard, exit 2, UI-identical). Bad RRULE / timezone fall to the server, which maps them to `BAD_REQUEST` so the CLI exits 2 — `schedules-service.ts` converts the three validator errors to `TRPCError({ code: "BAD_REQUEST" })`, the same service-boundary pattern `egress-rules-service.ts` uses.
- **`enable`/`disable` wrap the single `toggle` mutation idempotently.** The router only flips `spec.enabled`, so the CLI reads current state and toggles only when the target differs; an already-matching state is a no-op success. The read-then-toggle race is accepted (solo-CLI assumption).
- **`reset-session` matters only for continuous schedules.** The CLI reads the schedule first: a fresh-mode schedule has no accumulated binding, so it reports "nothing to reset" and exits 0 without enqueuing the durable outbox event. The `get` precheck also yields the not-found classification, since the server's `resetSession` returns `void` and silently no-ops on an unknown id rather than throwing.

`schedules.createCron` is **not** wrapped — the UI abandoned cron creation and the CLI creates RRULE-only. Everything is tRPC (`schedules.list`, `get`, `createRRule`, `updateRRule`, `toggle`, `delete`, `resetSession`) — no REST surface and no service state beyond the per-host client. Agent refs resolve through the shared `AgentResolver` (`EXIT_AGENT_NOT_RESOLVED` (5)); schedule ids are opaque positional args. Unknown schedule id on `get`/`update`/`enable`/`disable`/`reset-session` exits the dedicated `EXIT_SCHEDULE_NOT_FOUND` (9), classified via `data.code === "NOT_FOUND"`; `delete` is idempotent (exit 0 on an unknown id), like `network revoke`.
